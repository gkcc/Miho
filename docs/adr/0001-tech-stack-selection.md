# ADR-0001：当前工具主线与实现形态

状态：已更新

## 背景

项目已经收敛到 ZZZ box / tier / 配队 / 培养 / 未来观察主线。当前目标不是完整桌面应用，而是先把本地可信规划闭环跑通。

## 约束条件

- 本地优先。
- 不读取登录态、账号标识或本地 profile。
- 不提交真实图片、真实 roster、数据库或本地运行产物。
- 公开社区统计只能作为参考信号。
- 缺失角色只能作为未来观察，不输出确定抽卡结论。

## 候选方案

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| Python CLI + JSON/Markdown | 修改快、测试轻、适合当前算法和报告验证 | 不是最终桌面体验 | 当前采用 |
| 完整桌面应用 | 体验好 | 当前会过早扩大范围 | 暂缓 |
| 纯文档规划 | 成本低 | 不能验证真实输入输出 | 不选 |

## 推荐结论

当前继续采用 Python CLI + JSON/Markdown 报告作为主线实现形态。

保留入口：

```text
tools/probes/prepare_zzz_meta_snapshot.py
tools/probes/extract_zzz_box_roster.py
tools/probes/build_agent_value_cards.py
tools/probes/run_zzz_box_value_pipeline.py
tools/probes/miho_probe_cli.py
```

## 原型阶段建议

- 先优化报告质量和评分解释。
- 先用本地 roster JSON 稳定算法。
- box 图识别只作为生成候选 roster 的辅助入口。
- 每次改动都要有单元测试或 CLI smoke 验证。

## 最终产品建议

最终桌面形态可以后置决策。等 box/tier 规划逻辑稳定后，再选择 UI 和本地存储方案。

## 不选择其他方案的原因

- 现在做完整桌面应用会分散算法和报告验证。
- 现在做复杂账号采集会带来隐私和维护风险。
- 现在重做数据库会提前锁死尚未稳定的数据模型。

## 后续验证任务

1. 提升账号内价值评分的可解释性。
2. 增加培养优先级的资源成本说明。
3. 增加当前队伍与未来观察队伍的置信度。
4. 增加 CLI 状态检查覆盖。

## 风险清单

- Prydwen 公开页面结构变化。
- 社区统计不能代表个人账号实际体验。
- box 图识别可能漏角色或误识别等级。
- 低等级高价值角色的培养成本容易被低估。
