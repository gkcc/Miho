# AGENTS.md

## 项目定位

本项目计划做一个本地优先的游戏练度跟踪与规划桌面应用，目标覆盖：

* 《绝区零》代理人练度、音擎、驱动盘、技能、邦布、终局活动完成情况。
* 《崩坏：星穹铁道》角色练度、光锥、遗器/饰品、行迹、星魂、终局活动完成情况。
* 基于当前 box、典型终局活动、培养资源、卡池与卫星信息，生成：

  * 阵容匹配建议；
  * 培养优先级；
  * 体力/开拓力规划；
  * 抽取建议。

最终产品形态应是一个卡片化、美观、可长期维护的桌面应用，而不是临时脚本或纯命令行工具。

## 总体原则

1. 本地可信数据闭环优先，AI 推荐靠后。
2. 数据采集必须可复现、可回放、可降级。
3. 不把 UI 自动点击、截图 OCR 作为主链路，只能作为最后兜底。
4. 不做外挂，不控制游戏客户端，不绕过登录风控，不绕过验证码，不高频请求。
5. 不上传用户 cookie/token，不把敏感信息写入日志、提交记录、报告或测试 fixture。
6. 推荐结论必须能解释：

   * 依赖了哪些数据；
   * 面向哪个终局目标；
   * 为什么这个角色、队伍或资源项优先；
   * 置信度是多少；
   * 哪些前提变化会推翻结论。
7. 不追求一次性大而全，优先跑通稳定闭环。

## 技术选型决策原则

ADR-0001 已完成技术选型调研并形成阶段性路线，但这不是无条件锁死最终产品技术栈。

后续技术验证必须围绕 ADR-0001 的结论展开，不再泛泛重复技术栈调研。若要推翻 MVP 主路线，必须通过新的 ADR 记录明确证据和原因。

技术验证仍必须同时评估：

1. 卡片化 UI 的开发效率和最终美观度。
2. 本地数据库、快照、diff、规划算法的可靠性。
3. 米游社 / HoYoLAB 数据采集的可行性和维护成本。
4. 打包体积、启动速度、内存占用。
5. 隐私保护和本地数据安全。
6. Codex / agent 修改代码时的可维护性。
7. 个人长期维护成本。
8. Windows 桌面使用体验。
9. 后续跨平台可能性。
10. 定时更新、桌面通知、后台任务支持情况。

ADR-0001 已评估的候选方案包括：

* Tauri + Rust + Svelte 或 React
* Flutter / Dart
* Python + PySide6
* C# + WinUI 3
* Rust + Slint / Iced / egui
* Electron + TypeScript

不允许只因为 Rust 性能强就默认选 Rust。当前暂定 Tauri + Rust 的原因是 UI 美观、隐私边界、本地数据库、快照 diff、规划算法、打包和 Codex 可维护性综合权衡后的结果。

后续决策必须继续区分：

* 原型阶段最适合什么；
* 最终产品阶段最适合什么；
* 数据采集阶段是否需要独立工具链；
* UI 是否应该和核心业务逻辑分离。

## ADR-0001 后的阶段性方向

当前阶段性方向是：

* Rust 负责核心业务逻辑、数据模型、快照 diff、规划算法、SQLite 访问边界和隐私脱敏。
* Svelte/Tailwind 负责卡片化、美观 UI 与信息架构验证。
* SQLite 负责本地数据。
* 数据采集 adapter 必须和 UI、规划算法、存储实现解耦。
* Python / TypeScript 可作为后续采集原型或 fixture 处理工具候选，但不得进入 MVP 主链路。

这仍是 MVP 阶段性路线，不是最终产品的永久锁定。任何改变都必须通过新的 ADR 说明。

## 数据来源分层

数据来源优先级从高到低：

1. 用户授权后可访问的官方战绩页面或接口返回数据。
2. 用户手动导出的 JSON / HTML / CSV。
3. 用户手动整理的截图或表格。
4. 公开攻略站、社区 wiki、官方公告、卡池公告。
5. UI 自动化截图 + OCR，仅作为兜底。

禁止：

* 绕过验证码、风控、加密、登录保护。
* 高频请求米游社 / HoYoLAB。
* 上传、打印、提交 cookie、token、账号标识。
* 把非官方爆料当成确定事实。
* 未经确认自动改写用户本地数据。
* 自动控制游戏客户端。

## 数据采集设计原则

不要优先实现“打开米游社 APP -> 点我的 -> 点游戏 -> 点战绩 -> 点角色详情”的完整自动点击采集流程。

例外：P0.6 允许验证米游社 APP 官方导出 / 分享图能力。该路线只能操作用户本地已登录 APP 内的官方 UI，不接底层 API，不读登录态，不抓包，不自动登录，不控制游戏客户端。

原因：

* UI 容易改版；
* 分辨率、缩放、主题、语言影响大；
* 登录状态和弹窗不可控；
* OCR 误差会污染练度数据；
* 很难做稳定回归测试。

当前探针验证优先级：

1. 米游社 APP 官方导出 / 分享图：用户本地 APP 已登录后，操作官方 UI 生成图片，再做 OCR / 图像识别 / 版面解析。
2. Visible UI Probe：用户手动进入页面后，只读当前窗口 UIA 控件树和可见文本，辅助判断按钮和字段可见性。
3. Local App Data Inventory：用户显式指定目录后，只读扫描非敏感元信息、JSON key、SQLite schema 和缓存形态。
4. 用户手动 JSON / HTML / CSV 导入。
5. 网页自动化或授权接口：必须另开 ADR，明确授权、脱敏、限流、失败处理和 Git 防泄漏。
6. 截图 + OCR：仅作为人工确认后的兜底，不自动入库。

OCR 结果必须人工确认后才能进入正式数据库。

## 米游社数据读取边界

米游社 APP 是社区和官方数据工具，不等同于游戏客户端。项目可以研究用户自己账号登录后已经可见的战绩和角色练度数据，但必须把“读取当前可见界面”“本地非敏感元信息盘点”和“登录态 / token / 网络层”分开处理。

### A. Visible UI Probe：允许

允许做只读可见界面探针，用于判断米游社 APP 当前窗口是否暴露可解析的角色练度文本。

允许范围：

* 用户手动打开米游社 APP，并手动进入角色概览或角色详情页。
* 工具只读当前窗口控件树和可见文本。
* 可以输出窗口标题、控件类型、控件名称、可见文本、控件边界和是否离屏等 UIA 元信息。
* 输出必须先脱敏，再写入 `data/probes/`。

禁止行为：

* 不点击。
* 不滚动。
* 不自动登录。
* 不抓包。
* 不读取 cookie/token/stoken/ltoken。
* 不写正式数据库。
* 不把 UIA dump 当作正式练度数据直接入库。

### B. Local App Data Inventory：允许，但必须显式本地只读

允许做本地 APP 数据目录 inventory，用于判断用户指定目录里是否存在非敏感、可解析的 JSON / SQLite / HTML / cache 数据来源。该行为只能由用户通过参数显式指定目录触发，工具不得自动猜测、遍历或读取 APP profile。

允许范围：

* 工具可以在用户指定目录下扫描米游社 APP 本地数据。
* 只列出文件路径、大小、mtime、扩展名、文件头、JSON key、SQLite 表名等元信息。
* 可以识别是否存在可能的 JSON / SQLite / HTML / cache 数据。
* 可以抽样读取非敏感结构化字段。
* 所有输出必须脱敏。
* 默认不输出大块原文。
* 输出写入 `data/probes/`，且不得提交 Git。

敏感文件限制：

* 默认不读取疑似 cookie/token/session/storage/login/auth 文件内容。
* 疑似敏感文件只记录“存在”和类型，不打印内容。
* 不保存真实账号标识、登录态文件内容、浏览器 profile 内容或 APP profile 内容。
* 不写 SQLite，不上传，不成为 MVP 主链路。

### C. Session / Token / Network：默认禁止，后续单独 ADR

以下能力默认禁止，除非后续单独 ADR 明确授权边界、存储策略、脱敏规则、限流、失败处理和 Git 防泄漏机制：

* 不读取明文 cookie/token/stoken/ltoken。
* 不打印、不保存、不提交登录态。
* 不抓包。
* 不绕过验证码、风控、加密或登录保护。
* 不高频请求米游社 / HoYoLAB。
* 不把授权接口调用、session 复用或网络采集混入 P0/P0.5 探针。

## P0.5：米游社 APP 可读性探针

ADR-0002 的 MVP 边界仍然有效。MVP 仍不实现自动点击米游社 APP，不实现自动登录，不读取真实 cookie/token，不写正式数据库，不把 OCR 自动入库，也不把浏览器自动化或 APP 自动点击作为主链路。

允许独立做两个只读 feasibility probe，用来验证米游社 APP 是否存在后续半自动采集价值：

* Visible UI Probe：用户手动进入角色概览或详情页后，只读当前窗口 UIA 控件树和可见文本。
* Local App Data Inventory：用户显式指定本地目录后，只读扫描文件元信息、JSON key、SQLite 表名和非敏感结构摘要。

这两个 probe 的目标是验证：

* 当前可见页面字段是否覆盖 ADR-0003 的本地数据模型草案；
* Windows UI Automation 是否能看到窗口、角色列表、角色名、详情文本和滚动容器；
* 用户指定目录里是否存在非敏感、可解析的 JSON / SQLite / HTML / cache 数据；
* 米游社 APP 页面是否疑似 WebView，是否存在后续 saved_html / browser_page adapter 的研究价值；
* 截图 / OCR 是否真的必要；
* 登录状态、弹窗、验证码、懒加载、缩放和 UI 改版风险是否会破坏可复现性。

这两个 probe 必须遵守：

* 不接入真实 cookie/token。
* 不读取疑似 session/token 文件内容。
* 不抓包。
* 不绕过验证码、风控、加密或登录保护。
* 不自动登录。
* 不实现完整自动点击流程。
* 不写正式数据库。
* 不把 OCR 结果自动入库。
* 不控制游戏客户端。
* 不成为 MVP 主链路。
* 不提交 `data/probes/` 结果。

spike 输出文档固定为：

```text
docs/spikes/0001-miyoushe-app-feasibility.md
```

该文档用于记录两个只读 probe 的运行方式、结果判定和人工 fallback，不是正式采集设计。若 probe 结果显示有半自动价值，后续必须另开任务或 ADR，再评估字段抽取 prototype、importer prototype、saved_html / browser_page adapter、人工截图字段对照或回到手动 JSON 导入。

米游社 APP 是官方社区和数据工具，不是游戏客户端。如果用户本地 APP 已登录，允许通过 RPA 操作米游社 APP 内的官方 UI，优先验证官方“导出 / 分享图”能力是否能提供稳定、可回放、可人工确认的练度数据来源。

允许范围：

* 自动点击米游社 APP 内的官方入口，例如：
  * 我的；

  * 战绩统计；

  * 导出 / 分享图；
  * 保存图片。
* 允许等待官方页面加载。
* 允许读取官方生成的分享图。
* 允许对分享图做 OCR / 图像识别 / 版面解析。
* 允许把识别结果输出为 probe JSON。
* 输出必须写入 `data/probes/`、`data/probes/exported_images/` 或 `data/probes/parsed/`，并不得提交 Git。

禁止范围：

* 不允许自动登录。
* 不允许输入账号密码。
* 不允许绕过验证码、风控、加密或登录保护。
* 不允许抓包。
* 不允许读取明文 cookie/token/stoken/ltoken。
* 不允许控制游戏客户端。
* 不允许把识别结果直接写入正式数据库。
* 不允许把真实导出图或 probe 结果提交 Git。

P0.6 仍然不是正式采集器。它只回答“官方导出 / 分享图是否足以支撑后续字段抽取 prototype”。若分享图字段覆盖不足，再回到 Visible UI Probe、Local App Data Inventory 或单独 ADR 评估授权接口。

## MVP 功能边界

MVP 不追求全自动。

MVP 最小闭环是：

1. 手动导入一份角色练度 mock 数据或样例数据。
2. 标准化为统一数据模型。
3. 保存本地快照。
4. 生成快照 diff。
5. 配置一个终局目标。
6. 生成当前 box 的目标缺口。
7. 生成培养优先级。
8. 生成体力/开拓力规划。
9. 用卡片化界面或报告展示结果。

MVP 暂不要求：

* 自动登录；
* 正式自动点击米游社 APP 完整采集流程，P0.6 官方导出 / 分享图 RPA 探针除外；
* 自动解析所有终局活动；
* 自动生成完美抽卡建议；
* 支持全部角色、全部遗器、全部驱动盘精细评分；
* 多账号云同步；
* 真正后台长期运行。

* 这个角色解决哪个终局痛点？

禁止：

* 只根据榜单推荐；
* 只根据厨力否定强度；
* 把未实装卫星当确定规划；
* 不看用户 box 直接建议抽。

## 推荐算法原则

阵容匹配评分不追求绝对准确，先做相对排序。

初版评分建议考虑：

* 角色是否拥有；
* 关键定位是否覆盖；
* 队伍核心是否完整；
* 练度是否达可用线；
* 是否符合敌人机制；
* 是否与另一队冲突；
* 培养成本；
* 角色泛用性；
* 用户当前资源压力。

输出必须包含解释，而不是只给分数。

示例输出结构：

```text
推荐队伍：
- A / B / C / D

适用目标：
- 某终局活动某一关

推荐原因：
- 已拥有核心角色 A、B；
- C 虽未毕业，但关键技能已达可用线；
- D 与另一队冲突较低；
- 当前缺口主要是 A 的技能等级和 B 的装备主词条；
- 预计投入 5~7 天体力可以达到可打线。

置信度：
- 中

不确定项：
- 当前 Boss 机制数据来自人工配置；
- 未读取游戏内最新阵容推荐；
- 装备副词条评分暂未精细化。
```

## 培养优先级规则

培养优先级按以下因素综合排序：

1. 终局收益。
2. 资源成本。
3. 泛用性。
4. 时效性。
5. 当前队伍是否从不可用变可用。
6. 是否影响多个终局目标。
7. 是否只是锦上添花。

优先级建议：

1. 能让一个终局队伍从不可用变可用的关键角色。
2. 多个队伍共用的泛用辅助或生存位。
3. 低成本高收益项：

   * 等级关键突破；
   * 核心技能；
   * 行迹 / 技能关键节点；
   * 武器 / 光锥关键等级。
4. 主 C 的关键装备和核心词条。
5. 副词条优化。
6. 只服务单一低收益场景的角色。

禁止默认建议：

* 把所有技能点满；
* 所有角色拉满；
* 为了极限评分无限刷遗器 / 驱动盘；
* 在未确认终局目标前给出抽卡结论。

## UI 产品原则

最终 UI 应是卡片化桌面应用。

核心页面建议：

1. 首页 Dashboard

   * 今日体力 / 开拓力建议；
   * 当前最重要培养目标；
   * 终局活动完成情况；
   * 抽卡风险提醒。

2. 角色页

   * 角色卡片；
   * 等级、武器 / 光锥、技能 / 行迹、驱动盘 / 遗器概览；
   * 与上次快照的变化。

3. 目标页

   * 终局目标卡片；
   * 推荐队伍；
   * 当前 box 匹配度；
   * 缺口项。

4. 规划页

   * 今日计划；
   * 本周计划；
   * 预计完成时间；
   * 资源投入优先级。

5. 抽卡页

   * 当前卡池；
   * 后续官方卡池；
   * 卫星 / 爆料低置信度信息；
   * 抽取建议；
   * 不抽风险；
   * 抽了是否有资源培养。

6. 设置页

   * 数据来源；
   * 账号配置；
   * 本地数据库位置；
   * 隐私与脱敏设置。

UI 禁止做成只有普通表格的管理后台。表格可以作为详情页辅助，但首页和主要页面必须以卡片为主。

## Subagent 使用原则

subagent 是开发协作方式，不是业务架构。

适合使用 subagent 的任务：

* 技术选型调研；
* 数据源可行性调研；
* UI 框架对比；
* 数据模型设计；
* 终局规则建模；
* 测试策略设计；
* 安全隐私检查。

不适合使用 subagent 的任务：

* 简单改名；
* 小 bug 修复；
* 单文件修改；
* 无需并行调研的小任务。

使用 subagent 时，必须要求：

1. 先只读分析。
2. 不要改代码。
3. 每个 subagent 输出结构化结论。
4. 主 agent 汇总而不是简单拼接。
5. 明确风险、置信度和推荐等级。

## 版本控制与远程推送

每次修改文档或代码并完成验证后，必须同步到远程仓库。

要求：

* 如果当前工作区是 Git 仓库且已配置远程，修改完成后必须提交 commit 并推送到当前分支对应的远程分支。
* 推送前必须先检查工作区状态，确认提交内容只包含本次任务相关文件，不夹带敏感信息或无关改动。
* 不允许提交或推送 cookie、token、stoken、ltoken、account_id、uid、浏览器 profile、APP profile、数据库文件或其他敏感数据。
* 如果当前目录不是 Git 仓库、未配置远程、没有权限推送，或用户明确要求本次不要推送，必须在最终回复中说明原因和当前文件状态。
* 推送失败时不得静默结束，必须报告失败原因和需要用户处理的事项。
